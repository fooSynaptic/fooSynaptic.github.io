一个生产级、可扩展的 LLM 生成服务完整架构 Case，用系统设计术语拆解，直接能当面试答案讲清楚。

🎯 需求背景

我们要设计一个面向企业用户的 LLM 问答服务（类似企业版 ChatGPT），支持：

  • 百万级并发请求，高峰时 QPS 可达 10k+
  • 流式响应输出（打字机效果）
  • 多模型路由：简单请求用 7B 小模型，复杂/长文本用 70B 大模型
  • 企业私有知识库 RAG、工具调用
  • 完整的安全、监控、容灾与可观测性

🏗️ 整体架构分层（从外到内）

1. 入口层：API Gateway（流量入口）

核心组件：Nginx/Envoy 反向代理 + Kong 网关

  • 功能：
    • 负载均衡：将请求分发到多个后端服务实例，避免单点瓶颈
    • 鉴权与限流：API Key/Token 验证，基于用户级别的速率限制（RPM/QPS 配额）
    • TLS 终止：统一处理 HTTPS 加密，减轻后端服务压力
    • 熔断与降级：后端服务异常时快速失败，返回预设降级响应

2. 请求路由层：分层模型路由 + 队列削峰

核心组件：智能路由服务 + Kafka/RabbitMQ 消息队列

（1）分层模型路由

根据请求特征自动分发到不同模型集群：

请求类型 路由规则 目标模型
简单问答/闲聊 短提示词、低复杂度意图识别 7B 轻量模型（低延迟、低成本）
复杂推理/代码生成 长上下文、多步骤任务 70B 大模型（高精度、高成本）
企业私有知识库问答 含企业 ID 标识、知识库标签 带 RAG 能力的专用模型实例

（2）队列削峰与异步处理

  • 同步请求：低延迟场景（<1s 响应）直接进入模型服务
  • 异步请求：高负载/长任务场景（如批量生成）写入消息队列，由 Worker 池消费处理
  • 作用：削峰填谷，避免突发流量压垮模型服务，实现流量平滑

3. 业务编排层：请求预处理与后处理

核心组件：Orchestration Service（编排服务）
这一层是业务逻辑的核心，串联起 RAG、工具调用、安全检查等所有非模型计算逻辑：

  • 请求预处理
    1. 提示词组装：根据用户请求、系统 Prompt、上下文历史拼接成最终 Prompt
    2. RAG 检索：调用向量数据库（如 Pinecone/Weaviate）检索企业私有知识库，将相关文档片段注入 Prompt
    3. 工具编排：解析用户请求中的工具调用指令（如查天气、查数据库），调用外部 API 并将结果注入上下文
    4. 安全护栏（Guardrails）：
      • 输入过滤：检测敏感词、恶意指令（如 prompt injection），拦截违规请求
      • 输入脱敏：替换请求中的用户隐私信息（手机号、身份证号）
  • 请求后处理
    1. 输出过滤：检测生成内容中的敏感信息、违规内容，替换或截断
    2. 格式转换：将模型输出转换为客户端需要的格式（JSON、Markdown、流式 SSE 等）
    3. 结果缓存：将相同请求的生成结果写入缓存，提升重复请求的响应速度

4. 模型服务层：核心推理集群

核心组件：模型推理服务 + 自动扩缩容 + 模型版本管理

  • 模型部署：
    • 基于 Triton Inference Server/vLLM 部署模型服务，支持动态批处理（Dynamic Batching）提升吞吐量
    • 7B 模型部署在低成本 GPU 节点，70B 模型部署在高规格 GPU（如 A100）节点，实现资源分层利用
  • 自动扩缩容：
    • 基于队列长度、GPU 利用率、推理延迟三个指标自动扩缩容实例数量
    • 高峰时自动扩容模型实例，低谷时缩容释放资源，降低成本
  • 流式交付：
    • 模型服务支持 SSE/websocket 流式输出，客户端实时接收生成的 token，实现打字机效果
    • 服务端维护会话状态，支持中断、续传、取消请求

5. 数据与可观测性层：日志、监控与反馈闭环

核心组件：Prometheus + Grafana + ELK Stack + 评估平台

  • 可观测性：
    • 指标监控：跟踪 QPS、延迟（TTFT/TPOT）、错误率、GPU 利用率、缓存命中率等核心指标
    • 日志收集：全链路日志采集，从请求入口到模型输出的完整链路追踪
    • 告警体系：设置延迟过高、错误率飙升、GPU 利用率异常等告警规则，实时通知运维人员
  • 缓存层：
    • 热点请求缓存：Redis 缓存高频相同请求的生成结果，降低模型服务压力
    • 会话上下文缓存：缓存用户会话历史，避免每次请求都从数据库读取
  • 评估反馈闭环：
    • 用户反馈收集：客户端支持对生成结果点赞/点踩，数据写入评估平台
    • 离线评估：定期抽样模型输出，用人工/自动化指标（如相关性、事实一致性）评估模型效果
    • 模型迭代：根据评估结果优化 Prompt 模板、调整路由策略，甚至触发模型微调

6. 安全与容灾体系

  • 数据安全：请求数据加密存储，模型输出脱敏，企业数据与公网模型隔离
  • 故障容灾：
    • 多可用区部署，单机房故障时自动切换到备用机房
    • 模型服务支持灰度发布，新版本模型上线时先小流量验证，异常时快速回滚
    • 关键路径降级：模型服务故障时，返回预设的兜底响应，保证服务可用性

✅ 面试亮点总结（按系统设计术语梳理）

  1. 分层架构：将流量入口、路由、业务编排、模型推理、可观测性分层设计，解耦各模块职责，便于独立扩展与维护
  2. 弹性扩缩容：基于队列削峰、自动扩缩容实现流量弹性，兼顾性能与成本
  3. 智能路由：分层模型路由策略，实现不同复杂度请求的资源最优分配
  4. 全链路可观测：从请求入口到模型输出的完整指标、日志、告警体系，保障服务稳定性
  5. 业务闭环：通过用户反馈与离线评估，实现服务的持续优化迭代
  6. 安全护栏：全链路的输入/输出过滤、数据脱敏、权限控制，保障企业级安全合规
