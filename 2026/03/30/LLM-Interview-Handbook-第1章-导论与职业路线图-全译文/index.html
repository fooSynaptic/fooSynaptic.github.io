第 1 章　导论、基础与 LLM 职业路线图

本章概述

开篇这一章希望在深入细节之前，先给读者一张地图。它从系统层面定义大语言模型是什么、现代 LLM 技术栈如何拼在一起、给出可实操的学习路线，并概括 GenAI 相关岗位招聘正在发生哪些变化。目的是让你在读到后续章节时，能明白「为什么书中要按 分词 → 嵌入 → 注意力 → 检索 → 适配 → 评测 → 服务」这样的顺序展开。

在一场扎实的面试里，候选人很少一上来就背架构名词。更常见的做法是先框定工作负载：模型在哪创造业务价值？周边系统（surrounding system）又怎样把「裸模型能力」变成可上线的产品？本章就是要帮助你建立这种框定问题的思维方式（framing mindset）。

Interview Anchor　面试锚点

维度 内容
面试官真正想测的 能否把 LLM 讲成工程系统，而不是孤立的科研热词。
高质量回答套路 先把 LLM 定义为「预训练的下一 token 预测模型」，再说明它如何嵌入更大的应用栈；把该栈与检索、提示、评测、服务、治理以及可量化的产品结果串起来。
常见低分答法 直接跳到具体模型名或媒体 hype。加分答法会交代：要解决的任务是什么、数据怎么走、有哪些可靠性控制、在灵活性 / 成本 / 风险之间如何权衡。

INTERVIEW CHEATSHEET　面试速记条

项 要点
要表达的亮点 LLM 不是整个产品；它只是推理与生成引擎，处于更大的「检索—工具使用—评测—交付」工作流之中。
最佳举例 解释为什么一个客服助手除了基座模型之外，还需要：提示设计、检索质量、监控、升级（escalation）规则与输出控制。
可延展方向 提及学习路线：从分词、注意力到 RAG、PEFT、服务、评测与治理。
资深候选人可加 把多模态、更小的专用模型、推理优化等行业趋势与真实产品取舍挂钩。
切忌这样说 把 LLM 工程等同于「只会 Prompt」，却不谈数据、质量度量与运维约束。

*译注：表中 Interview Cheatsheet 里的 red flag 是英语面试场景里的固定说法，指「危险信号 / 明显减分点」——面试官一听就觉得你视角偏了的那种表述；不是中文里「红旗」一词的其它含义。*

基础：LLM 究竟是什么

大语言模型是在极大规模上训练、用于预测序列中下一个 token 的神经网络。这个目标看似简单，却能让模型内化关于句法、事实、结构、风格与任务行为的统计规律——训练语料规模巨大。

但在面试中，只说到「会生成文本」往往不够。更应再往上抽象一层：LLM 的价值在于可被嵌入到各类工作流中，用于分类、检索、摘要、基于工具的推理，以及结构化输出草稿等。

正因如此，本书后续才会从分词与嵌入，一路讲到检索、适配、提示、评测与服务。这些层次并不是为了「方便学习」而硬凑在一起，而是决定一套 GenAI 系统是否有用、有依据、够快、够安全、经济上可持续的真实运行分层。

图 1.1　在现代 LLM 技术栈上学习与面试的实用路线图

下面的路线图把整本书整理成一串循序渐进的学习层（learning layers）。它帮助你一眼看到：为什么靠前的章节必须先讲「机制」，而靠后的章节才把笔墨放在检索、适配与部署上——而不是在基座模型概念处就收官。

路线图之所以要紧，是因为太多候选人还没能把底层机制讲清楚，就急着跳到最时髦的话题。书中给出的更稳妥顺序，是先把机制层面的东西夯实，再抬升到产品形态、评测与部署叙事。

原书在图中将六大块排成两行：上行自左至右为 Layer 1～3，从文本、Token、上下文走到嵌入与注意力，再到预训练与模型家族；下行自左至右为 Layer 5、4、6（注意编号与原图一致），分别是适配、PEFT 与评测，检索、RAG 与提示，以及服务、治理与产品化。这不是随意贴的六个标签：Layer 1～3对应「模型与数据如何被表示、如何被训练出来」；Layer 4～6对应「如何把模型嵌进检索与提示流水线、怎样在资源与风险约束下做适配与度量、怎样交付并治理整条产品」。

图下角左右两条说明，原书英文分别是 Use this as a study plan: mechanics first, system design second, production trade-offs third，以及 Hiring loops often test these later layers most because they reveal engineering judgment——直述过来就是：把这张图当学习计划，顺序是「先机制 → 再系统设计 → 最后才谈生产权衡」；面试与招聘里往往更爱追问后面几层，因为那才看得出你有没有工程判断，而不是只会背前面的名词。

LLM 与 GenAI 岗位的学习路线

对多数工程师而言，最有效的路线是分层的，而不一定是按年份线性排布：

  1. 先掌握文本基础与模型机制；
  2. 再理解检索如何改变上下文质量；
  3. 然后学习适配与服务怎样让系统可上生产；
  4. 在此基础上，再专精某一方向：评测、Agent、多模态、安全或垂直 Copilot 等。

同一套分层视角也适用于简历与面试叙事：你可以清楚说自己最强的是「检索 + 评测」，还是「服务 + 优化」，或是「把 Agent 工作流产品化」。这比笼统声称「我什么都懂」要可信得多。

下文表 1.1用来把全书锚定在当前产业方向上——它不是 hype 词汇表；每一行都对应团队如何招人、如何定项目边界、如何区分技术深度。

请把趋势表当作优先级过滤器：真正的区分度往往不在于又多刷了一个泛泛的模型教程，而在于能否讲清检索质量、评测、推理约束，以及**人在回路（human in the loop）**应在何处保留。

表 1.1　对工程学习路线与面试预期影响最大的 LLM 趋势

趋势 为何重要 强候选人应能讨论什么
更长的上下文窗口 更多内容能塞进 prompt，但无关上下文仍会伤害答案与成本。 为何在模型已支持很大窗口时，检索、排序、上下文压缩仍然关键。
多模态系统 许多企业与消费场景下，纯文本已非默认。 图像、音频或文档输入如何改变评测、延迟与体验设计。
更小的专用模型 团队常在前沿质量 vs. 成本 / 可控性 / 部署灵活性之间做权衡。 何时选用更小的任务型模型、PEFT 或路由（routing），而不是永远调用最大模型。
评测与治理 产品规模化后，信任与度量比 demo 更难。 离线 / 在线评测、幻觉控制、护栏、升级路径与监控。
推理优化 成本与延迟越来越多地决定 LLM 产品是否成立。 量化、批处理、缓存、结构化输出与输出 token 预算纪律。
工具使用与 Agent 真实产品往往把语言模型与 API、数据库、工作流引擎结合。 规划 vs. 执行、工具选择、状态管理与人机协同控制。

图 1.2　本书的思维导图（高阶索引）

这张思维导图刻意保持高层，为的是给你一张「章节之间如何衔接」的可视化目录：后面读到 PEFT、服务、多模态等小节时，你应当仍觉得它们是同一条系统叙事上的延伸，而不是一条条彼此无关的面试冷知识。

请注意导图的运动方向：它是从**表示（representation）移向系统（systems）**的。也就是说，能把机制层的理解与产品行为、生产约束对接起来的回答，在现场会更有说服力。

原书中心结点写作 LLM Interview Mastery（书中全大写 Mastery），可译为**「LLM 面试所需的全貌掌握」或简写为「LLM 面试掌握」**——指的就是把整本书各块捏成一块可讲述的整体。围绕该中心，原图用示意位置挂出若干分支（名称以原书为准，下附汉译）：

  • Tokens and Context — Token 与上下文
  • Embeddings and Retrieval — 嵌入与检索
  • Attention and Model Design — 注意力与模型设计
  • RAG and Grounding — RAG 与 Grounding（保持通用译名）
  • Fine-Tuning and PEFT — 微调与 PEFT
  • Serving and Decoding — 服务与解码
  • Governance and Deployment — 治理与部署
  • Multimodal Systems — 多模态系统

以上分支与上一节六层路线图同一套书内逻辑，只是导图更强调「从基础概念簇到交付与治理」的整体形状。

加餐：LLM / GenAI 岗位的简历结构建议

Bonus Layer（加餐层）

一份有竞争力的「大模型方向」简历，读起来应像一份工程系统说明，而不是热词拼贴。招聘方想看到的是你对真实工作负载的 ownership 证据：评测、检索质量、Agent 编排、可靠性、安全与可量化影响。

这部分放在第一章，是因为职业定位应与本书教授的同一套栈对齐：你做了什么、如何度量、你承担了哪些权衡。

表 1.2 把「LLM 简历该长什么样」落成可检查的章节清单。

板块 应证明什么 推荐写法模式
标题与摘要 与岗位的清晰对齐 一句话定位，例如「构建生产级 LLM、RAG 与评测系统的 AI 工程师」，再加 3–4 个高价值专长。
核心技能 技术深度，而非堆关键词 按 LLM 栈 / 检索栈 / 服务栈 / 云与可观测 / 评测分组，而非随机罗列工具。
经历 bullet 可度量的 ownership 先写构建的系统，再写决策或优化，最后量化相关性、延迟、成本、可靠性或采纳度。
项目 自驱与完整闭环 至少一个旗舰项目：架构、评测闭环、失效控制与部署形态。
公开信号 市场可信度 GitHub、技术文章、演讲或产品——且与同一叙事相互印证时再写。

「强候选人」一句话（原书措辞意译）

「LLM 产品是围绕模型的一整套系统；因此最强的那批候选人，能讲清从分词与嵌入，到检索、适配、服务、评测，以及商业上每一层存在的理由。」

表 1.3　如何把经历 bullet 写得像「工程证据」

档次 示例
偏弱 「参与了聊天机器人与检索流水线。」
更好 「为内部支持助手搭建 BM25 + 向量的混合检索流水线，在评测中提高了有依据命中，并减少了幻觉型回答。」
更好（带约束与指标） 「设计 BM25 + 向量检索 + 重排 + 引用校验的内部支持助手流水线，有依据答案命中率提升约 18%，升级工单量下降，且中位响应时间仍满足产品 SLO。」
为何有效 同时体现系统设计、决策质量、可量化结果与工程约束管理。

读者可带走的三点

  1. LLM = 系统里的引擎，检索、评测、服务与治理与「会不会写 Prompt」同等重要。
  2. 学习/叙事尽量按「机制 → 系统 → 生产」分层，再选专精方向。
  3. 简历与口述尽量用事实、指标与权衡说话，避免空泛热词。

按原书顺序，第 2 章标题为 Tokens, Tokenization, and Context Windows，将讨论 Token 是什么、为何不等同于词、BPE / SentencePiece、上下文窗口、超长输入如何处理、特殊 Token，以及生产里如何做 token 预算等。
